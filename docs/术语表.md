## A — Add-in（插件 / 加载项）

**大白话**：一个让某个软件多出新功能的"外挂"。

**一句话定义**：被宿主软件（如 WPS / Word）在启动时加载的一段程序代码, 通过宿主提供的 API 增加功能或修改行为。

**在本项目里指**：

- `Local_Wps_Vsto.MyAddin` — 原版「公文助手」的 Add-in 主类（提供 30+ 个 ribbon 按钮）
- `A_GongwenPatcher.Connect` — 我们自己写的 Patcher Add-in（不提供功能, 只在内存里 hook 原版的 IsVip 函数）

**生活类比**：装在浏览器上的"广告屏蔽插件"就是 Add-in。本身不能单独运行, 必须挂在浏览器里。

**想再深入**：[原理分析.md](原理分析.md) §一 §二

## A — AddinsCL（Crash List, WPS 加载失败黑名单）

**大白话**：WPS 内部记录"哪个 Add-in 启动时崩过"的一个清单。崩了 3 次以上就会自动禁用它。

**一句话定义**：WPS 注册表项 `HKCU:\Software\Kingsoft\Office\WPS\AddinsCL`, 用来累积统计每个 Add-in 启动失败的次数。

**在本项目里指**：

- 我们的 Patcher Add-in 在每次 OnStartupComplete 成功后, 会自动从 AddinsCL 把自己条目清掉, 防止偶发崩溃累积导致被永久禁用
- 这是 [踩坑全集.md](踩坑全集.md) #11 教会的反向防御机制

**生活类比**：类似 Win10 任务管理器里的"启动影响 高"标记 —— 系统自动统计, 累积太多就自动处理。

## A — AddinsWL（Whitelist, WPS 加载白名单）

**大白话**：WPS 默认对来历不明的 Add-in 警惕, 加在白名单里的才会被自动加载。

**一句话定义**：WPS 注册表项 `HKCU:\Software\Kingsoft\Office\WPS\AddinsWL`, 我们必须把自己的 ProgId 写进去。

**在本项目里指**：

- 我们的 install.ps1 会写 `Local_Wps_Vsto.MyAddin = ""` 和 `A_GongwenPatcher.Connect = ""` 两条, 让 WPS 默认加载
- **写入时必须只 add property, 不能 New-Item -Force**, 否则会清空别人的条目（[踩坑全集.md](踩坑全集.md) #20）

## A — AppDomain

**大白话**：.NET 程序里一个隔离的"沙盒"。同一个 .exe 进程内可以有多个互不干扰的 AppDomain。

**一句话定义**：.NET CLR 提供的进程内逻辑隔离单元, 每个 AppDomain 有独立的 Assembly 加载列表 / 静态字段 / 安全策略。

**在本项目里指**：

- WPS 启动时拉起 mscoree.dll → 创建一个 AppDomain → 把 GongwenPatcher.dll 和 Local_Wps_Vsto.dll 都加载到**同一个 AppDomain**里
- 同一 AppDomain 内 Patcher 才能 hook Local_Wps_Vsto 的方法 — 跨 AppDomain 不能用 Harmony

**生活类比**：一栋楼里的不同套间, 都在同一栋楼（进程）里, 但内装互不影响。

---

## C — CLSID

**大白话**：每个 COM 对象的 32 个字符的"身份证号"。

**一句话定义**：Class ID, COM 系统用来唯一标识一个 COM 类的 GUID, 形如 `{9F1A2B3C-4D5E-6F70-8190-A1B2C3D4E5F6}`。

**在本项目里指**：

- Patcher 的 CLSID 写死在 `src/GongwenPatcher/Connect.cs` 的 `[Guid("9F1A2B3C-...")]` 特性
- WPS 通过这个 CLSID 找到「应该用 mscoree.dll 加载哪个 .NET 类」

**生活类比**：身份证号。每个 COM 类全球唯一一个, 不会撞车。

**想再深入**：[原理分析.md](原理分析.md) §三

## C — COM (Component Object Model)

**大白话**：微软 90 年代发明的一套"让不同语言写的程序能互相调用"的机制。

**一句话定义**：Windows 上的二进制级跨语言对象互操作框架, 现今被 .NET、UWP、UIAutomation 等大量沿用。

**在本项目里指**：

- WPS 是用 C++ 写的, 通过 COM 加载用 C# 写的 Add-in
- COM 对象通过 CLSID 注册在 HKCR / HKCU / HKLM 的 `\Software\Classes\CLSID\{...}` 下

**生活类比**：联合国翻译机 — 让说不同语言的人能互相听懂。

---

## C — CodeBase

**大白话**：告诉 .NET "这个 dll 在硬盘上具体哪个文件夹"。

**一句话定义**：注册表 `InprocServer32` 子键下的字符串, 形如 `file:///C:/Users/.../GongwenPatcher.dll`, 用来告诉 .NET 加载器去哪找这份 dll。

**在本项目里指**：

- 我们用 CodeBase 加载 → 不需要把 dll 注册到 GAC（全局程序集缓存）
- 这是为什么本项目**不需要管理员权限**就能装 — 所有路径都在用户目录下

**为什么重要**：
- GAC 注册需要 sn.exe + 强名 + 管理员权限, 反而容易触发杀软
- CodeBase 只是改个 HKCU 注册表 + 拷贝 dll 到用户目录, 完全平民化

---

## D — dnlib / dnSpyEx

**大白话**：能直接把 .dll 拆开重新装回去的瑞士军刀。

**一句话定义**：开源 .NET PE 文件读写库 / GUI 工具, 能查看、修改、保存 .dll 的 IL 字节码。

**在本项目里指**：

- 我们**没用** dnlib 直接改原 dll, 因为改完 PublicKeyToken 会变, 触发杀软
- 改用 Harmony 在内存里改 — dnlib 只在调研期试过

**为什么不用**：
- 改完 dll 字节 SHA256 变化 → 杀软启发式必然报警
- Harmony 路线 dll 字节零修改, 风险低很多

---

## G — GAC (Global Assembly Cache)

**大白话**：Windows 系统级的 .NET dll 仓库, 装到这里所有进程都能直接 import。

**一句话定义**：`%windir%\Microsoft.NET\assembly\` 下的全局缓存, 由 sn.exe 强名签名后的 dll 才能注册进去。

**在本项目里指**：

- 原版「公文助手」装到了 HKLM + GAC, 强名签名 PublicKeyToken=475a36b05c42bd98
- 我们**不动 GAC** — 我们的 Patcher 注册到 HKCU, 用 CodeBase, 平级共存
- 所以**保留**原版 GAC 版 Local_Wps_Vsto.dll, Patcher 在它启动后再 hook IsVip

**为什么不动 GAC**：
- 改 GAC 需要管理员权限
- 改 GAC 强名签名 dll 字节会变, 微软 PE 校验会报错
- 改 GAC 下的 dll 杀软几乎必报

---

## H — Harmony

**大白话**：能在 .NET 程序运行起来后, "偷偷换掉"某个函数实现的工具。

**一句话定义**：开源 C# 库（pardeike/Harmony）, 在 AppDomain 内动态修改 IL, 把目标方法的入口替换为我们的 Prefix / Postfix / Transpiler。

**在本项目里指**：

- 我们用 Harmony 把 `Local_Wps_Vsto.UserUtil.IsVip()` 和 `HasLogin()` 这两个方法的 IL 在内存里替换成 `return true`
- 修改只在内存里发生 — 硬盘上的 dll 文件**字节零变化**, SHA256 不变

**为什么用 Harmony 而不是直接改 dll**：

| 项目 | 改 dll | Harmony |
|---|---|---|
| dll 字节是否改变 | 改变 | **不变** |
| SHA256 是否变 | 变 | **不变** |
| PublicKeyToken 是否变 | 变（强名失效） | **不变** |
| 杀软触发率 | 高 | **低** |
| 卸载是否完全 | 难 | **WPS 关掉就回原样** |

**生活类比**：演员替身 — 演员还是那个演员, 但某个特定动作换了个人来演, 录像放完替身就走了。

**想再深入**：[Patcher方案.md](Patcher方案.md) 全文

## H — HKCU / HKLM / HKCR / Wow6432Node

**大白话**：Windows 注册表的几个不同分区, HKCU 只影响当前用户, HKLM 影响整台电脑。

**一句话定义**：

- `HKCU` = HKEY_CURRENT_USER, 当前用户的设置, 不需要管理员权限就能改
- `HKLM` = HKEY_LOCAL_MACHINE, 整台机器的设置, **需要管理员权限**
- `HKCR` = HKEY_CLASSES_ROOT, COM 类注册的视图（HKCU + HKLM 的合并）
- `Wow6432Node` = 32 位程序在 64 位 Windows 上看到的"虚拟视图", 64 位程序不会看到这里

**在本项目里指**：

- 我们的 Patcher 写 HKCU 下的 4 组键, **不动 HKLM**
- 但 WPS 是 32 位进程 → 需要在 HKCU 下的 `Software\Classes\CLSID` **和** `Software\Classes\Wow6432Node\CLSID` 都注册一份, 否则 32 位 WPS 看不到（[踩坑全集.md](踩坑全集.md) #21）

**生活类比**：

- HKCU = 你自己房间的钥匙
- HKLM = 整栋楼的钥匙
- Wow6432Node = 给 32 位"老房客"看的注册副本

## I — IL（Intermediate Language, 中间语言）

**大白话**：.NET 编译出来的"半成品代码", 不是机器码, 也不是源码, 由 CLR 运行时再翻译成机器码。

**一句话定义**：Common Intermediate Language, .NET 标准 ECMA-335 定义的栈式虚拟机字节码, .dll 文件里实际存放的就是 IL 字节。

**在本项目里指**：

- 反编译 = 把 IL 翻译成 C#（人类可读）
- Harmony patch = 在内存里把某个方法的 IL 字节替换成另一段（如 `ldc.i4.1; ret` 表示"压栈 1, 返回"）
- IsVip() 原本的 IL 是查文件 / 查注册表 / 比对日期, patch 后只剩 2 个字节

**生活类比**：JPEG 是"压缩好但还没渲染的图像数据", IL 是"编译好但还没翻译成 CPU 指令的代码"。

**想再深入**：[原理分析.md](原理分析.md) §四

## I — ILSpy

**大白话**：把 .dll 拆回 C# 源码的免费工具。

**一句话定义**：开源 .NET 反编译器（icsharpcode/ILSpy）, 输入一个 .dll, 输出可读的 C# 源码（方法体级别）。

**在本项目里指**：

- 我们用 ILSpy 7.2.1 把 `Local_Wps_Vsto.dll` 拆出 60+ 个 .cs 文件, 放在 `src/Local_Wps_Vsto_v2/`
- 整个反编译过程**自动化**, 见 `tools/decompile.ps1`

**生活类比**：把已经做好的菜倒回去, 还原成菜谱（不是 100% 完美, 但够人类读懂）。

## J — JS Add-in (WPS 自家的 JavaScript 加载项)

**大白话**：WPS 自己发明的"用 JavaScript 写插件"的方案。

**一句话定义**：WPS 12.0+ 提供的 wpsjs 框架, 用 publish.xml + ribbon.xml + main.js 三件套写插件, 通过 Edge WebView2 内的 V8 引擎解释执行。

**在本项目里指**：

- 我们试过用 JS Add-in 替代 COM Add-in（路线 B）
- 但 WPS 12.1.0.16910+ 个人版**禁掉了 dev 模式加载**, 个人版只能加载官方应用商店审核过的插件
- 详见 [踩坑全集.md](踩坑全集.md) #26

## L — Local_Wps_Vsto

**大白话**：原版「公文助手」插件的核心 dll 名字。

**一句话定义**：原作者「公文助手」团队编译产出的 .NET 程序集, 文件名 `Local_Wps_Vsto.dll`, 是 WPS Office 的 COM Add-in。

**在本项目里指**：

- 我们用 ILSpy 反编译这份 dll, 重命名 namespace 后的源码在 `src/Local_Wps_Vsto_v2/`
- 我们的 Patcher 启动后会去找内存里加载好的 `Local_Wps_Vsto`, 然后 hook 它的 IsVip 等方法

## M — mscoree.dll

**大白话**：Windows 自带的"启动器", 让 .NET 代码能在普通进程里跑起来。

**一句话定义**：Microsoft .NET CLR Execution Engine, 是 .NET COM Add-in 注册时填到 InprocServer32 (default) 的内容。

**在本项目里指**：

- 我们的 Patcher 注册时, InprocServer32 的 default 值就是 `mscoree.dll`
- WPS 拉起这个 Add-in 时实际是先加载 mscoree.dll, mscoree 看到 RuntimeVersion=v4.0.30319 + Class=GongwenPatcher.Connect, 才创建 .NET AppDomain 把我们的代码拉起来

## P — PE (Portable Executable)

**大白话**：Windows 上的 .exe / .dll 文件格式标准。

**一句话定义**：Microsoft 定义的可执行文件格式, 包含 Header / Section / Import Table / IL（如果是 .NET dll）等。

**在本项目里指**：

- 我们的 verify.ps1 用 `Assembly.ReflectionOnlyLoadFrom` 读 dll 的 PE 字节, 用 IL Reader 找 IsVip 的字节签名, 校验是否 patch 成功
- Harmony patch 不改 PE 文件本身, 只改内存里 JIT 出的方法表

---

## P — ProgId

**大白话**：CLSID 的人类可读别名。

**一句话定义**：Programmatic Identifier, 形如 `A_GongwenPatcher.Connect` 或 `Local_Wps_Vsto.MyAddin`, 通过注册表关联到 CLSID。

**在本项目里指**：

- WPS 的 AddinsWL 白名单里写的是 ProgId 而不是 CLSID
- Word 的 Addins 注册表项名也是 ProgId

**生活类比**：CLSID 像身份证号, ProgId 像姓名。

---

## P — PublicKeyToken (PKT, 强名 token)

**大白话**：.NET dll 的"防伪标签", 8 字节, 表明这份 dll 是哪个开发者签的。

**一句话定义**：从 dll 的强名 RSA 公钥派生出的 8 字节哈希, 改 dll 字节就会变 / 失效。

**在本项目里指**：

- 原版 Local_Wps_Vsto.dll 的 PKT = `475a36b05c42bd98`
- 我们的 Patcher 故意做成**无强名**（PKT=null）, 只在 HKCU 弱命名加载, 与原版共存不冲突
- 我们**不动**原 dll, 它的 PKT 保持 `475a36b05c42bd98`, GAC 还能继续用

## R — Ribbon

**大白话**：Office / WPS 顶部那条带按钮的"菜单栏"。

**一句话定义**：Microsoft Office 2007 引入的图形化命令带, 由 ribbon.xml 描述结构, 业务程序通过 IRibbonExtensibility 接口提供 callback。

**在本项目里指**：

- 「公文助手单机版2.4.1」就是一个 ribbon tab
- 它的 XML 在 `src/Local_Wps_Vsto_v2/Local_Wps_Vsto/Resource1.resx` 里嵌入
- callback（按钮按下后调谁）通过 `MyAddin.cs` 内的 OnAction1 / OnAction2 ... 实现
- **WPS 12.1+ 屏蔽 onAction 回调** — 这是项目天花板, [踩坑全集.md](踩坑全集.md) #25

## R — RuntimeVersion

**大白话**：告诉 .NET "这个 dll 是 .NET 哪个版本编译的"。

**一句话定义**：注册表 InprocServer32 子键的 RuntimeVersion 值, 形如 `v4.0.30319`, 必须与 dll 的目标框架对得上。

**在本项目里指**：

- 我们的 Patcher 是 net48 编译, RuntimeVersion 写 v4.0.30319

---

## S — Strong Name (强名签名)

**大白话**：给 dll 盖个"防伪章", 改一个字节章就废了。

**一句话定义**：用 RSA 私钥对 dll 内容做签名, 任何人改 dll 字节都会让签名失效。

**在本项目里指**：

- 原版 Local_Wps_Vsto.dll 是强名签名的（PKT=475a...）
- 我们的 Patcher 是**弱名 / 无签名**, 故意如此, 因为 GAC 不需要弱名
- 改强名 dll → PKT 失效 → GAC 加载失败 → 业务整挂, 所以我们绝不直接改原 dll

## V — VSTO (Visual Studio Tools for Office)

**大白话**：用 C# / VB.NET 写 Office 插件的官方框架。

**一句话定义**：Microsoft 的 Office 集成开发框架, 基于 COM Add-in + .NET 包装层, 提供 ribbon / event handler 等托管 API。

**在本项目里指**：

- 原版「公文助手」是用 VSTO 写的, 文件名 Local_Wps_Vsto 就是这个意思
- 我们的 Patcher 走的是裸 COM Add-in 路线, 不依赖 VSTO 运行时

## W — wpsjs

**大白话**：WPS 自家的 JS Add-in 命令行开发工具。

**一句话定义**：金山 WPS 提供的 npm 包 `wpsjs`, 提供 `wpsjs init` `wpsjs debug` 等命令, 用于在本地开发 / 调试 JS Add-in。

**在本项目里指**：

- 我们走过路线 B（JS Add-in）, 用 wpsjs debug 部署 publish.xml 到 `%APPDATA%\kingsoft\wps\jsaddons\`
