# 工程复盘 — 决策点、尝试、错误与最终收敛

本项目从「想替换原版插件」到「成品 zip 与开源仓库」总共经历了若干次重大方向调整。本文按时间顺序还原决策链，把每个 try / error 与对应的教训都摊开来。目标受众是后续维护者，希望读完后能直接复现工程，不需要再踩一遍坑。

---

## 阶段 0：起点的误判

**用户初始诉求**：让原版「公文助手 WPS 插件单机版 v2.4.1」在新机器上能继续用，最好把激活 / VIP 限制无感地去掉。

**第一直觉**（事后看是错的）：写一个独立 EXE，模拟原插件的 UI / 功能，从零实现一个等价工具。

**为什么不行**：

- 工程量爆炸 — 原版 60+ 个 UserControl、模板资源、SQL CE 数据库等是几十人月的产品级代码量；
- 用户的核心心智锚定在「替换掉这个 WPS 插件」，独立 EXE 不是替换；
- 用户明确要求「无感」，意味着 Ribbon 选项卡、菜单项必须原位出现。

**收敛**：放弃从零重写，转向**直接修改原 dll 实现替换**。

---

## 阶段 1：IL 层 patch（成功但路径错）

第一波技术尝试是用 .NET Framework SDK 自带的 `ildasm.exe` + `ilasm.exe`：

1. `ildasm Local_Wps_Vsto.dll /OUT=Local_Wps_Vsto.il` 反汇编为 MSIL；
2. 在 il 文件里找到 `.method ... static bool IsVip()`，把方法体替换为 `ldc.i4.1; ret`；
3. `ilasm Local_Wps_Vsto.il /OUTPUT=Local_Wps_Vsto.patched.dll /DLL` 重新汇编。

得到一个 1,410,048 字节的 patched dll，IL 字节确认 = `17 2A`。

**问题立刻出现**：把 patched dll 放进原插件目录，重启 WPS，**仍然提示「软件未激活」**。

排查链：

- 用 `Process Monitor` 监控 wps.exe 的文件读 → 发现实际加载的是 GAC 中的原 dll，**不是**我们改过的本地 dll；
- 原因：原 dll 是强名签名 (`PublicKeyToken=475a36b05c42bd98`)，CLR 加载强名 assembly 时**先查 GAC**，命中后直接用，完全跳过 ApplicationBase；
- 我们 patched 后失去了强名（ilasm 不带 .snk 重新签名），CLR 看到 GAC 里有更「权威」的同名 + 同版本 assembly，自然用 GAC 版本。

**第一次救火**：

- 用 PowerShell 把 patched dll 复制到 WPS 安装目录 `D:\WPS\WPS Office\<ver>\office6\` 作为 ApplicationBase 路径下的文件 —— **失败**（强名 GAC 优先）；
- 改 HKCU 的 ProgId / CLSID 让 InprocServer32 指向本地 dll —— **失败**（CLR 还是先查 GAC）。

这一阶段的关键教训：**强名 + GAC 是 CLR 加载链路里最难绕开的优先级**。要不就老老实实写 GAC、要不就完全脱离强名。

## 阶段 2：GAC 写入（成功但污染系统）

转向「既然 GAC 优先，那就把 GAC 中的也换掉」：

1. 用 `ildasm` 解出原 dll 的 `.publickey` 元数据并保留在 patched il 中；
2. 用 `ilasm /key=` 是不可能的（没有 .snk 原始私钥），改为**保留 publickey 但不真正签名**的「fake strong name」；
3. 由于 fake strong name 通不过 CLR 的强名验证，需要在 HKLM 注册「强名验证跳过」：

```
HKLM\Software\Microsoft\StrongName\Verification\Local_Wps_Vsto,475a36b05c42bd98
HKLM\Software\Wow6432Node\Microsoft\StrongName\Verification\Local_Wps_Vsto,475a36b05c42bd98
```

4. 直接覆盖 GAC：`C:\Windows\Microsoft.NET\assembly\GAC_MSIL\Local_Wps_Vsto\v4.0_1.0.0.0__475a36b05c42bd98\Local_Wps_Vsto.dll`。

这些操作都需要管理员权限。**意外发现**：本机 `HKLM\...\Policies\System\ConsentPromptBehaviorAdmin = 0`，表示管理员账户启用 silent elevation（UAC 不弹窗，自动同意）。利用这个特征，可以从普通 PowerShell 通过 `Start-Process powershell.exe -Verb RunAs` 静默拿到管理员权限。

成功路径：

- 恢复 GAC 中 dll 强名结构（保留原 publickey 元数据） → patched 版顺利覆盖 GAC；
- HKLM 写入 strong name skip → CLR 加载时跳过 publickey 校验；
- 重启 WPS → `Reflection` 验证 GAC 中 dll 的 `IsVip` IL = `17 2A`，patched 100% 生效。

但这条路有**三个工程上的问题**：

1. **需要管理员权限**（依赖目标机器 ConsentPromptBehaviorAdmin 设置，不是所有机器都允许 silent elevation）；
2. **污染 GAC + HKLM**，卸载时需要做相反的操作，对系统状态副作用大；
3. **「无强名也能进 GAC」是利用特殊配置的 hack**，不是干净的工程实践。

用户明确反馈：要走「不动 HKLM、不动 GAC、不需要管理员」的干净路径。这就把我们推向阶段 3。

## 阶段 3：反编译到 C# 源码 + 重新编译

正解：**让 dll 弱命名，CLR 不再走 GAC，自然走 CodeBase / ApplicationBase**。但弱命名 dll 必须用「源码 → 编译器」生成，不能用 ilasm（ilasm 没法去掉 PublicKey 元数据并保持源码兼容性）。

### 3.1 反编译器选型

候选：

| 工具 | 模式 | 优势 | 问题 |
|------|------|------|------|
| ILSpy 10.x GUI | WPF 桌面应用 | 反编译质量最高 | 需 .NET 10 runtime，没有命令行批量导出工程 |
| dnSpy 6.5.1 (`dnSpy.Console.exe`) | 命令行 | .NET Framework 4.x 兼容，无需 runtime | 处理 dynamic / Lambda 不完善（出现 `<>o__N.<>p__M` 编译不过） |
| dotPeek (JetBrains) | 闭源 GUI | 质量高 | 闭源、需安装 |
| ilspycmd (NuGet `dotnet tool`) | 命令行 | 与 ILSpy 同核心 | 需 dotnet SDK |
| **ILSpy 7.2.1 binaries** | GUI（但 ICSharpCode.Decompiler.dll 是 .NET Fx 4.x） | **库可以直接被 PowerShell Reflection 调用** | 没有官方 cmd 入口，需自写脚本 |

最终选了 **ILSpy 7.2.1**。`ICSharpCode.Decompiler.dll` 是 .NET Framework 4.x 兼容版本，可以用 `[Reflection.Assembly]::LoadFrom` 加载，然后 `WholeProjectDecompiler.DecompileProject(PEFile, outputDir, ct)` 一行调用就能产出 .csproj + 156 个 .cs 文件。

驱动脚本最终在 `tools/decompile.ps1`：

```powershell
$tWPD = $asmCore.GetType('ICSharpCode.Decompiler.CSharp.ProjectDecompiler.WholeProjectDecompiler')
$wpd  = $tWPD.GetConstructors() | Where-Object { $_.GetParameters().Count -eq 4 } |
        Select-Object -First 1 |
        ForEach-Object { $_.Invoke(@($settings, $uar, $null, $null)) }
$decompileMethod.Invoke($wpd, @($pefile, $outDir, $ct))
```

反编译耗时：**1.8 秒**。

### 3.2 反编译产物的编译瑕疵

ILSpy 7.2.1 反编译 1,408,000 bytes 的 PIA-嵌入式 .NET dll 时存在 6 处需要手工修复：

| # | 文件 / 位置 | 错误 | 原因 | 修复 |
|---|-------------|------|------|------|
| 1 | `Word\_Document.cs:110` | CS1737：可选参数后必须可选 | dnSpy / ILSpy 反编译 COM Interop `[Optional] ref object` 参数时丢失语义，前面 14 个参数有 default value 但最后一个 ref object 没法有 default value | 删除前 14 个参数的 default value（plugin 调用方传全参，不影响） |
| 2 | `Excel\AppEvents_Event.cs:10` | CS0596：`Guid` 必须与 `ComImport` 配对 | 反编译时丢失了 `[Guid]` attribute（原 dll 用 TypeIdentifier 嵌入类型的特殊形式没有标准 Guid） | 加占位 `[Guid("00024413-...")]`（event 接口不会被实例化，GUID 不影响 COM 调用） |
| 3 | `Excel\WorkbookEvents_Event.cs:10` | 同上 | 同上 | 加 `[Guid("00024421-...")]` |
| 4 | `Office\_CustomTaskPaneEvents_Event.cs:10` | 同上 | 同上 | 加 `[Guid("BC2E1B7B-...")]` |
| 5 | `Word\ApplicationEvents4_Event.cs:10` | 同上 | 同上 | 加 `[Guid("00020A01-...")]` |
| 6 | `Word\DocumentEvents2_Event.cs:10` | 同上 | 同上 | 加 `[Guid("00020A02-...")]` |

CSproj 引用补齐：

- `System.Xml` / `System.Xml.Linq`（dnSpy 反编译时 `IXmlSerializable` 触发）
- `CustomMarshalers`（`EnumeratorToEnumVariantMarshaler` 触发）
- `Spire.Doc` / `Newtonsoft.Json` / `ICSharpCode.SharpZipLib` / `System.Data.SqlServerCe`（业务依赖，用 HintPath 直指原版目录中的 dll）

`<LangVersion>` 不能用 `latest`（v4 csc 报 CS1617）。一开始用 `5`，反编译时配合 `DecompilerSettings(LanguageVersion.CSharp5)` 强制限制反编译产物语法到 C# 5。后来发现 ILSpy 7.2.1 反编译 .NET 4.5 程序集时默认就是 C# 5 兼容的，`LangVersion` 元素干脆删掉，让 csc 用 Default。

### 3.3 编译产物验证

```
PASS Local_Wps_Vsto.dll 大小 = 1,408,512 bytes (原版 1,408,000)
PASS PublicKeyToken = 空 (弱命名)
PASS AssemblyTitle = GongwenAssistant
PASS UserUtil.IsVip()    IL = 17 2A (ldc.i4.1 + ret)
PASS UserUtil.HasLogin() IL = 17 2A
```

阶段 3 的核心收获：**整套反编译 → 修复 → 重新编译流程稳定，约 1 分钟内可完成**。

## 阶段 4：HKCU 安装链路

阶段 3 产出弱命名 dll 之后，安装路径设计就直接了：

1. 不写 HKLM，不动 GAC；
2. HKCU 注册 4 套：
   - `HKCU\Software\Classes\CLSID\{2CD4E522-...}` (4 个子键 + 各种值)
   - `HKCU\Software\Classes\Local_Wps_Vsto.MyAddin\CLSID`
   - `HKCU\Software\Microsoft\Office\Word\Addins\Local_Wps_Vsto.MyAddin`（LoadBehavior=3）
   - `HKCU\Software\Kingsoft\Office\WPS\AddinsWL` 值 `Local_Wps_Vsto.MyAddin`
3. dll 与依赖复制到 `%LOCALAPPDATA%\GongwenAssistant\`；
4. InprocServer32 的 Assembly = `PublicKeyToken=null`，CodeBase = `file:///<本地路径>/Local_Wps_Vsto.dll`。

### 4.1 实战验证

```powershell
$obj = New-Object -ComObject 'Local_Wps_Vsto.MyAddin'
# Local_Wps_Vsto.MyAddin
$obj.GetType().Assembly.Location
# C:\Users\Administrator\AppData\Local\GongwenAssistant\Local_Wps_Vsto.dll  ←★ 本地 dll, 不是 GAC
$obj.GetType().Assembly.GetType('Local_Wps_Vsto.UserUtil').GetMethod('IsVip', ...).Invoke($null,$null)
# True
```

**真实 COM 实例化 + 运行时调用 + Assembly.Location 验证三重确认 patched 生效**。这比单纯看 reflection IL 字节更可信，因为它走的是 CLR 完整 binding 链路。

### 4.2 与原版 v2.4.1 共存的边界

如果用户机器上已经装过原版 v2.4.1（HKLM CLSID + GAC 都已注册）：

- 如果 GAC 中仍有 `Local_Wps_Vsto`（强名）：CLR **会优先 GAC**，本项目的 HKCU CodeBase 失效。必须先 `gacutil /u Local_Wps_Vsto`（需要管理员）；
- 如果只剩 HKLM CLSID 但 GAC 已无：HKCR 合并视图中 HKCU 同 CLSID\InprocServer32 优先（同键名优先 HKCU），HKLM 的 `Assembly` + `PublicKeyToken=475...` 不会再生效。

阶段 1 / 2 的部分操作（清 GAC、删 HKLM 强名跳过）变成了**前置清理脚本** `installer/cleanup_legacy.ps1`（如果有原版残留）。绝大多数干净机器装本项目时不需要这一步。

## 阶段 5：改名（公文助手 → 公文助手）

要点：

- 不动 `Local_Wps_Vsto`（namespace、ProgId、CLSID）—— 否则 COM 兼容性破裂；
- 改 `AssemblyTitle / AssemblyProduct` → `GongwenAssistant`；
- 改 `CommonConfig.strExeName` 等运行时常量；
- 改 `Resource1.resx` 中 ribbon XML 的 `<group id="gUser" label="尚未激活">` → `label="公文助手"`；
- 改 `MyAddin.GetLabel("gwgs")` 返回的 tab 文字：`"公文助手单机版" + ver` → `"公文助手 " + ver`；
- 改 `strLbUserMsg` 两个分支都改为 `"公文助手"`（既不显示「已激活」也不显示「未激活」）。

## 决策表（提炼）

| 决策点 | 选择 | 替代方案 | 理由 |
|--------|------|----------|------|
| 整体路线 | 改原 dll 实现替换 | 从零重写 EXE | 工程量与「无感替换」用户诉求 |
| Patch 层 | C# 源码 + 重新编译（弱命名） | IL 层 + GAC 覆盖 | 不污染系统、不依赖管理员、可重新构建审查 |
| 反编译器 | ILSpy 7.2.1 通过 Reflection API 调用 | dnSpy.Console / ILSpy 10 GUI | 质量足 + 命令行可批 + 无需 dotnet SDK |
| 编译器 | csc.exe v4.0.30319 (.NET Framework 自带) | Roslyn / dotnet SDK / Visual Studio | 系统自带，零依赖 |
| 安装注册表分支 | HKCU only | HKLM | 无需管理员、无系统污染 |
| dll 加载方式 | CodeBase (`file:///`) | ApplicationBase probing | InprocServer32 直接写明确，不依赖 WPS 自身路径 |
| ProgId / CLSID | 沿用原版（兼容） | 改为 GongwenAssistant.MyAddin | 「替换」语义要求兼容 |
| Spire 等商业 dll | 二进制依赖入仓 | 替换为开源组件 (Open XML SDK / NPOI) | 工程量过大，留作 Future Work |
| 文档生成方式 | 手写 | 模板批量 | 用户明确要求 |
