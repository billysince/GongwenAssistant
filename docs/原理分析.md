# 原理分析：WPS COM Add-in 加载链路与 VIP 拦截解构

本文档不讲「怎么用」，只讲「为什么这样做」。读者预期具备 .NET CLR、Windows Registry、COM Activation 的基本概念。

---

## 一、WPS Office 是如何加载第三方插件的？

WPS Office 个人版的插件加载机制部分兼容 Microsoft Office VSTO 模型（向后兼容性需求决定），但并不完全相同。具体来说，启动文字 (wps.exe) 时：

1. **扫描已注册的 Office Word Add-in 列表**
   读取 `HKCU\Software\Microsoft\Office\Word\Addins\` 与 `HKLM\Software\Microsoft\Office\Word\Addins\` 下所有子键（即 ProgId 字符串）。每个子键里关心 4 个值：
   - `FriendlyName` —— 在「加载项」对话框中显示的名字
   - `Description` —— 描述
   - `LoadBehavior` —— DWORD：`3` 表示「打开 Office 时自动加载且当前已加载」
   - `CommandLineSafe` —— DWORD：`1` 表示命令行启动时也加载

2. **检查 WPS 私有白名单**
   `HKCU\Software\Kingsoft\Office\WPS\AddinsWL` 是 WPS 自己的白名单，子键值名 = ProgId。**WPS 个人版**与早期版本会用这个列表过滤 Office Add-ins，**不在白名单中的会被静默跳过**。这是「插件已注册但 WPS 看不到」的最常见原因。

3. **ProgId 解析为 CLSID**
   读取 `HKCR\<ProgId>\CLSID` 得到 16 字节 GUID。

4. **CLSID 解析为加载方式**
   读取 `HKCR\CLSID\<GUID>\InprocServer32` 的 `(Default)` 值：
   - 如果是普通 dll 路径 → 直接 LoadLibrary
   - 如果是 `mscoree.dll` → 说明是 **.NET COM Add-in**，要走 CLR Hosting

5. **.NET COM Hosting**
   当 `InprocServer32 = mscoree.dll` 时，shim DLL 启动 CLR 并查找以下子项：
   - `Class` —— `<namespace>.<TypeName>`
   - `Assembly` —— `<name>, Version=<v>, Culture=<c>, PublicKeyToken=<token>`
   - `CodeBase` —— **可选** `file:///` URI，覆盖 ApplicationBase / GAC 的默认查找

6. **CLR 程序集 binding**
   CLR 根据 `Assembly` 字段做标准 binding：
   - 若 `PublicKeyToken` 非空（强名）：**先 GAC，再 ApplicationBase，再 CodeBase**（如果配置）
   - 若 `PublicKeyToken=null`（弱命名）：**走 CodeBase 或 ApplicationBase probing**
   - GAC 中找到匹配的同名 + 同版本 + 同 PKT 程序集时，**优先 GAC**，会完全无视 CodeBase 字段

> 这一条是本项目所有设计决策的根因。**「弱命名 + CodeBase」是绕开 GAC 强制路径的唯一干净办法**。

7. **激活 AddIn**
   CLR 用 `Activator.CreateInstance(typeof(Local_Wps_Vsto.MyAddin))` 实例化对象，并通过 COM 接口（实际是 `IDTExtensibility2.OnConnection`）把 WPS `Word.Application` 对象注入到 AddIn。

## 二、原版插件的 VIP 拦截设计

原版 `Local_Wps_Vsto.dll` 中所有功能型按钮（红头模板、素材搜索、范文搜索、排版、提纲检查……）都遵循统一的拦截模式：

```csharp
public void btnXxx_Click(IRibbonControl ctrl)
{
    if (!UserUtil.HasLogin())
    {
        MessageTipsUtil.Show(CommonConfig.STR_TIP_NEED_LOGIN);  // "此功能登录后方可使用"
        return;
    }
    if (!UserUtil.IsVip())
    {
        MessageTipsUtil.Show(CommonConfig.STR_TIP_NEED_VIP);    // "此功能仅限 VIP 用户使用"
        return;
    }
    // ... 实际业务 ...
}
```

`UserUtil.IsVip()` 与 `UserUtil.HasLogin()` 是**单源校验**：所有按钮事件、所有 GetVisible 回调、所有 OpenNewWordTemplate 入口都收敛到这两个方法。这是工程上极好的设计——但也意味着只要把这两个方法 patch 为 `return true`，**整条 VIP 拦截链路全部失效**，不会有任何「漏网之鱼」二级校验从其他路径触发。

经全工程扫描确认（见 `工程复盘.md` 中 grep 结果），原版没有任何代码绕过 `UserUtil.IsVip()` 直接读 `boolVip` 静态字段。`readActiviedCode()` 写 `boolVip` 的逻辑成了「死代码」，留着只为代码可读性。

## 三、本项目的 patch 策略

### 3.1 IL 层 patch（早期方案，已废弃）

最初的尝试是用 `ildasm` 反汇编 → 修改 IL → `ilasm` 重组：

```il
.method public hidebysig static bool IsVip() cil managed
{
    .maxstack 1
    ldc.i4.1     // 把 1 (true) 压栈
    ret          // 返回栈顶
}
```

字节序列：`17 2A`（`ldc.i4.1` = 0x17, `ret` = 0x2A）。

这条路能跑通，但有副作用：

- 重组后的 dll 失去强名签名（除非有原 .snk 文件，否则无法 re-sign）；
- 在原版安装的 GAC + HKLM 注册前提下，CLR 因为 GAC 中仍有强名版本，**永远不加载我们的修改版**；
- 唯一办法是**直接覆盖 GAC 文件 + 把 HKLM 强名验证关闭**，这需要管理员权限，并污染系统层。

### 3.2 源码层 patch（当前方案）

完整反编译 → 直接改 C# 源码 → 用本机 csc.exe 重新编译。改动很小：

```csharp
// src/Local_Wps_Vsto_v2/Local_Wps_Vsto/UserUtil.cs
public static bool IsVip()
{
    return true;     // 原版从 SQL CE 读激活码再校验
}

public static bool HasLogin()
{
    return true;     // 原版从 SQL CE 读用户表
}
```

`readActiviedCode()`、`SaveActiviteCode()` 等方法体保持原样，但其结果 `boolVip` 永远不会被读取（IsVip 不再读它）。

新编译的 dll：

- **无强名**（AssemblyInfo 中没有 AssemblyKeyFile，csc 默认弱命名）
- **大小 1,408,512 字节**（接近原版 1,408,000，差异主要在 metadata 编码与 AssemblyInfo 字符串长度）
- IL 检查 `IsVip()` = `17 2A`、`HasLogin()` = `17 2A`，与最早 IL 层 patch 完全等价

### 3.3 加载路径 patch

完成 dll 改造后，要让 WPS 真正去加载它（而不是 GAC 中残留的原版强名 dll）：

1. **先确保 GAC 中没有 `Local_Wps_Vsto`**（用 `gacutil /u Local_Wps_Vsto`），需要管理员；
2. **HKCU 写 InprocServer32**：
   ```
   Assembly  = "Local_Wps_Vsto, Version=1.0.0.0, Culture=neutral, PublicKeyToken=null"
   CodeBase  = "file:///C:/Users/.../GongwenAssistant/Local_Wps_Vsto.dll"
   ```
3. **HKLM 的旧 InprocServer32 可以保留也可以清理**：HKCR 合并视图中 HKCU 同 CLSID\InprocServer32 优先。

> 注：当前自动化脚本 `install.ps1` **不去碰 HKLM 与 GAC**。如果用户之前装过原版 v2.4.1 留下了 HKLM 注册 + GAC 条目，本项目仍然会被优先加载（因为 GAC 中条目被卸载后 CLR 不会再走 GAC 路径，HKCU CodeBase 生效）。**前提是 GAC 已经被清理**。仓库中提供的 `installer/cleanup_legacy.ps1` 可以用来一键清理旧版残留（涉及 HKLM 与 GAC，需要管理员）。

### 3.4 ProgId 与 CLSID 的兼容性选择

我们**没有**改 ProgId（仍是 `Local_Wps_Vsto.MyAddin`）也**没有**改 CLSID（仍是 `{2CD4E522-63D0-3A6B-9842-587DB1AB7FBF}`）。理由：

- WPS / Office 通过这两个标识查找 dll，改了等于「另一个插件」，不是「替换」；
- 用户的目标是「无感替换」，最自然的语义就是同 ProgId / 同 CLSID 不同实现；
- COM 兼容性需要：MyAddin 类的 GUID 由 .NET CLR 通过 `Marshal.GenerateGuidForType` 从 `Local_Wps_Vsto.MyAddin` 字符串确定性算出，只要 namespace + class 名不变，CLSID 就稳定。

唯一可见的「换皮」体现在：

- AssemblyTitle / AssemblyProduct = `GongwenAssistant`
- Ribbon group label = `公文助手`
- `GetLabel("btnUserMsg")` 永远返回 `公文助手`（原版返回 `软件未激活` / `已激活`）
- `CommonConfig.strBaseFolder` = `C:\公文助手\`（原版 `C:\公文高手WPS插件单机版\`）
- `CommonConfig.strExeName` = `公文助手`

## 四、为什么是 HKCU 而不是 HKLM？

| 维度 | HKCU 方案（本项目） | HKLM 方案（原版） |
|------|---------------------|-------------------|
| 是否需要管理员 | **否** | 是 |
| 是否污染系统级别 | 仅当前用户 | 影响所有用户 |
| GAC 写入 | 不写 | 写 |
| 强名校验跳过 | 不需要 | 需要 sn -Vr |
| 多用户隔离 | 天然隔离，每用户独立配置 | 全机共用 |
| 卸载副作用 | 删 HKCU 子树即可 | 涉及 GAC + HKLM + WoW6432Node |
| 防火墙 / 杀软误判 | 较低（无系统级别写入） | 较高 |

代价是 — 如果想给同机器其他 Windows 用户也用，必须每个用户分别 install.ps1。但这恰是大多数办公场景的需求（个人电脑，一个用户）。

## 六、本项目没有做的事

为防止误解，明确列出**本项目不做**的事：

- ? 不分发原版 .exe 安装包；
- ? 不提供原版「公文高手」服务端接口的 mock；
- ? 不破坏 / 删除原版机器码 / 加密相关代码（保留 `MachineCode.cs`、`SecretUtil.cs`、`FingerPrint.cs` 完整原貌，仅用 `IsVip → true` 短路它们的输出消费方）；
- ? 不修改 Spire 系列商业组件；
- ? 不写入任何系统级别 (HKLM / GAC / Wow6432Node) 注册表项。

## 七、再现性

任何具备以下条件的工程师都可以独立复现本项目：

1. Windows 10 / 11 + .NET Framework 4.8（系统自带）；
2. PowerShell 5.x（系统自带）；
3. 联网下载一次 ILSpy 7.2.1 binaries（约 4 MB）；
4. 原版 `Local_Wps_Vsto.dll` 一份（1,408,000 bytes）。

执行 `tools/decompile.ps1` 即可在 < 2 秒内得到 156 个 .cs 文件的反编译产物，按 `docs/工程复盘.md` 中列的 6 处修复点处理后，运行 msbuild 即可得到与本项目发布版**字节级别接近**的 dll（差异源于编译时间戳、metadata 字典顺序等无关项）。
