## [#01] WPS 加载链路完全靠猜

**问题**：项目开端，对"WPS 个人版加载 COM Add-in 走哪些注册表"一无所知，照搬 Microsoft Office VSTO 经验失败。

**初始假设**：跟 Office Word 一样，写 `HKCU\Software\Microsoft\Office\Word\Addins\<ProgId>`，`LoadBehavior=3`，应该就够了。

**实际表现**：注册了，WPS 启动后看不到选项卡，连"加载项"对话框里都没出现。

**验证步骤**：

```powershell
# 1. 看 WPS 实际启动时读了哪些注册表项
Process Monitor 过滤 wps.exe + RegOpenKey/RegQueryValue
# → 发现 wps.exe 还查询了 HKCU\Software\Kingsoft\Office\WPS\AddinsWL
# → AddinsWL 是 KEY 不是 VALUE_NAME, 每个值的"名字"=ProgId, "值"通常为空字符串
```

**根因**：WPS 个人版有**自己的白名单 `AddinsWL`**，不在白名单中的 COM Add-in 被它跳过，即便 Office Word Addins 那一套都注册了也没用。这是 WPS 跟 Office 的关键差异。

**修正**：`installer/install.ps1` 中显式写入 `HKCU\Software\Kingsoft\Office\WPS\AddinsWL`，键名 = ProgId、值 = 空字符串。同时尝试写 `HKCU\Software\Kingsoft\Office\Common\AddinsWL` 兼容老版本 WPS。

**教训**：跨 Office 兼容产品别 100% 信任 Microsoft 文档，要把目标进程的真实行为 Trace 一遍。

## [#02] IL patch 后 dll 装上去不生效

**问题**：用 `ildasm` + `ilasm` 在 IL 层把 `IsVip()` 改成 `ldc.i4.1; ret`，重启 WPS，**「软件未激活」仍然显示**。

**初始假设**：是不是 `MyAddin.MyAddInInitialize` 在启动时缓存了 `IsVip` 返回值？

**验证**：

```powershell
# 在 PowerShell 中 New-Object ComObject 'Local_Wps_Vsto.MyAddin' 后, 反射读 UserUtil.IsVip 的 IL
$obj = New-Object -ComObject 'Local_Wps_Vsto.MyAddin'
$asm = $obj.GetType().Assembly
Write-Host $asm.Location           # ← 关键
```

**发现**：`$asm.Location` 输出的是 GAC 中的 dll：

```
C:\Windows\Microsoft.NET\assembly\GAC_MSIL\Local_Wps_Vsto\v4.0_1.0.0.0__475a36b05c42bd98\Local_Wps_Vsto.dll
```

不是我们改过的 `D:\工作\...\Local_Wps_Vsto.dll`。

**根因**：原 dll 是强名签名 (`PublicKeyToken=475a36b05c42bd98`)。CLR 加载强名 assembly 时：

1. 先查 GAC：`fusion.dll` 算出 `<name>__<token>` 目录，命中则用 GAC 版本；
2. 不命中再走 ApplicationBase / CodeBase。

只要 GAC 中有同名 + 同版本 + 同 PKT 的 dll，**ApplicationBase 永远不会被走到**。

**临时修正（不对，但顺手记下来）**：当时尝试把 patched dll 拷贝到 WPS 安装目录 `D:\WPS\WPS Office\<ver>\office6\`，企图让 ApplicationBase probing 找到它。**没用**，原因同上。

**最终修正**：见 [#03] 与 [#10]。

**教训**：CLR binding 链路里**强名 + GAC** 是优先级最高的。要绕开它只有两条路：要么用强名版（写 GAC 或保留同样的 PKT 跳过验证），要么**编译产物根本就是弱命名**。

## [#03] 用强名跳过验证 + 覆盖 GAC（成功但污染系统）

**思路**：既然 GAC 优先，那就**把 GAC 里那份替换掉**。但我没有原 .snk 私钥文件，不能用同样的 PKT 重新签名。

**hack 路径**：

1. 反汇编时**保留**原 dll 的 `.publickey` 元数据（不是 `.publickeytoken`，是完整的 publickey blob）：

   ```il
   .publickey = (00 24 00 00 04 80 00 00 ... )   // 160 字节, 原 dll 中复制
   ```

2. `ilasm` 重新组装。此时 dll 在 metadata 层有完整 publickey，但**没有真正的签名 hash**——CLR 加载时会发现 hash 不匹配。

3. 在 HKLM 注册**强名验证跳过**：

   ```
   HKLM\Software\Microsoft\StrongName\Verification\Local_Wps_Vsto,475a36b05c42bd98
   HKLM\Software\Wow6432Node\Microsoft\StrongName\Verification\Local_Wps_Vsto,475a36b05c42bd98
   ```

   两个键都要写（64 位 view + 32 位 view 都覆盖）。

4. `gacutil /i patched_Local_Wps_Vsto.dll`，把 patched dll 写进 GAC（或者直接 `Copy-Item` 覆盖 GAC 目录下的文件）。

**陷阱 #3.1**：第一次跑时漏了 32 位 view 的注册项 (`Wow6432Node`)，结果 32 位 wpp.exe 子进程仍然校验失败。WPS 启动后会 fork 一个 32 位 `prome` 后台进程，**32 位 / 64 位强名验证跳过键必须双写**。

**陷阱 #3.2**：第一次跑 `gacutil /u Local_Wps_Vsto` 失败，因为 patched dll 还没有真签名，GAC 拒绝识别，无法卸载。解决：先把**原 dll**塞回 GAC，再 `gacutil /u`，然后再 `gacutil /i patched`。

**陷阱 #3.3**：HKLM 写不进去 — 提示"拒绝访问"。本机 PowerShell 是用户态的，没 admin。`Start-Process powershell -Verb RunAs` 又会弹 UAC 对话框，RDP 环境下没法点。

**意外发现**：检查 `HKLM\Software\Microsoft\Windows\CurrentVersion\Policies\System\ConsentPromptBehaviorAdmin` = `0`，意味着管理员账户用户启用了 **silent elevation**（UAC 自动同意不弹窗）。这是该机器特殊配置，不通用。

```powershell
# 在 silent elevation 下, RunAs 不会弹 UAC
Start-Process powershell.exe -Verb RunAs -ArgumentList '-File C:\Temp\elevate.ps1' -Wait
```

**临时收益**：阶段 3 这条路真的跑通了，patched dll 进了 GAC，HKLM 跳过验证，WPS 加载时确认 IL 是 patched 版本。

**为什么不用这条**：

- 依赖 `ConsentPromptBehaviorAdmin=0`，普通机器没这设置就要弹 UAC；
- 污染 HKLM + GAC，卸载麻烦；
- 真签名跳过是利用 Windows 信任机制的 hack，杀软容易飘红。

**最终修正**：放弃 IL patch 路线，转向**编译出弱命名 dll** + **HKCU CodeBase 加载**（[#10]）。

## [#04] PowerShell GBK 控制台与 UTF-8 脚本

**问题**：写好的 PowerShell 脚本里有中文字符串（路径、日志），在子进程 PowerShell 中执行时变乱码或解析错。

**例 1**：`Resolve-Path 'D:\工作\20260520\...'` 在新进程里返回 `D:\工作\20260520\...` 看似正常但路径校验失败。Test-Path 返回 False。

**例 2**：

```powershell
Start-Process powershell.exe -ArgumentList '-File', 'D:\工作\20260520\xxx.ps1' -Verb RunAs
```

子进程加载 .ps1 时按 ANSI(GBK) 解析，但文件本身是 UTF-8（无 BOM），中文字符串乱码。

**根因**：

- Windows 控制台默认输入输出代码页是 **936 (GBK)**，不是 65001 (UTF-8)。
- PowerShell 5.x 读 .ps1 文件时，**没有 BOM 的 UTF-8 文件被默认按 ANSI(GBK) 解析**。
- 跨进程传递参数时，参数字节流又走系统 ANSI 一次，二次乱码。

**修正**：

1. 所有 .ps1 文件用 **UTF-8 with BOM** 保存，让 PowerShell 5.x 识别为 UTF-8（PowerShell 7.x 默认 UTF-8 无问题）。
2. 关键 .ps1 开头加：

   ```powershell
   $OutputEncoding = [System.Text.Encoding]::UTF8
   [Console]::OutputEncoding = [System.Text.Encoding]::UTF8
   ```

3. **永远不让中文路径作为命令行参数**穿越 `Start-Process` 边界——改用 ASCII 中转目录 `C:\Temp\gw_xxx\`，把需要的文件先复制过去再调用。
4. `installer/install.ps1` 设计上**只接受 ASCII 路径作为 InstallDir**，默认 `%LOCALAPPDATA%\GongwenAssistant`。

**教训**：在中文 Windows + RDP 场景中，**所有要跨进程的内容必须 ASCII 化或显式 UTF-8 BOM**。

## [#05] EnumChildWindows 在 WPS 上崩溃

**问题**：尝试用 `EnumChildWindows` 列出 WPS 主窗口的所有子窗口，目的是定位 Ribbon "公文助手" tab 的 hWnd，模拟点击切换 tab。

**现象**：PowerShell 子进程跑 `EnumChildWindows` 后退出码 `4294967295`，整个 host shell 崩溃。

**初始假设**：是不是 P/Invoke 签名错？

**验证**：在简单的 Notepad 上跑同样的代码，正常返回十几个子窗口。

**根因**：WPS 个人版 UI 是 **Qt 自绘窗口**，大多数控件不是真正的 Win32 子窗口，`EnumChildWindows` 走到某些 Qt 内部代理时触发 Access Violation 把整个 PowerShell host 进程崩了（cursor 的 shell 工具就报 exit_code: 4294967295）。

**修正**：彻底放弃 UIAutomation / EnumChildWindows / SendInput 这条 GUI 自动化路线。WPS Qt 自绘控件**不能**用标准 Win32 自动化工具操作。

转向：

- **反射验证**：`Assembly.ReflectionOnlyLoadFrom` 读 PE 文件 IL 字节，验证 `IsVip()` IL = `17 2A`。
- **COM 实例化验证**：`New-Object -ComObject 'Local_Wps_Vsto.MyAddin'` + `$obj.GetType().Assembly.Location` + 反射调用 `UserUtil.IsVip()`，证明 CLR 真的从指定路径加载了我们要的 dll、`IsVip` 真的返回 true。

这两条路加在一起比 GUI 自动化更可靠，**而且不依赖窗口可见 / 可点击**，纯逻辑层面验证。

**教训**：UI 自动化在 RDP + Qt + DPI 缩放下基本不可用。**业务正确性应该从 API 层而非 UI 层验证**。

## [#06] dnSpy.Console 反编译 dynamic 调用失败

**问题**：第一次反编译用 `dnSpy.Console.exe` (.NET Framework 4.x 兼容版)，输出 159 个 .cs 文件，但 `msbuild` 编译报 **592 个错**。

**错误分布**：

- `MesBoxShow.cs:62` 等多处含 `<>o__13.<>p__0 = CallSite<Action<CallSite, object, string>>.Create(...)` —— C# `dynamic` 调用编译器生成的隐藏类型，dnSpy 没还原成 dynamic 语法；
- `Excel\_Application.cs` `[IndexerName]` 属性应用位置错误；
- `Excel\AppEvents_Event.cs` 等 `[ComImport]` 接口缺 `[Guid]`；
- `Word\_Document.cs:140` `[Optional] ref object` 与前面 14 个 default value 参数冲突 (CS1737)。

**根因**：

- dnSpy 反编译质量在 C# 4+ 的 `dynamic` / Lambda / 匿名类型上不够好；
- PIA 嵌入类型（TypeIdentifier）反编译时 metadata 缺失，没有 `[Guid]`；
- `[Optional] ref` 是 COM Interop 特殊形式，C# 编译器不认可它作为 optional。

**修正**：

放弃 dnSpy.Console，换 **ILSpy 7.2.1** 的 `ICSharpCode.Decompiler.dll`（也是 .NET Framework 4.x 兼容版本）。ILSpy 内部反编译引擎质量明显更高：

- `dynamic` 还原成 `dynamic d = obj; d.Speak(...)`；
- Lambda / 匿名类型还原干净；
- 不输出 `<>o__N.<>p__M` 编译器隐藏标识符。

ILSpy 7 GUI 模式没有 `ilspycmd` 命令行，但 `ICSharpCode.Decompiler.dll` 的 API 可以**直接被 PowerShell Reflection 调用**：

```powershell
$asm = [Reflection.Assembly]::LoadFrom('ilspy72\ICSharpCode.Decompiler.dll')
$wpd = $asm.GetType('ICSharpCode.Decompiler.CSharp.ProjectDecompiler.WholeProjectDecompiler')
$decompiler = $wpd.GetConstructors()[1].Invoke(@($settings, $resolver, $null, $null))
$decompiler.GetType().GetMethod('DecompileProject', 3).Invoke($decompiler, @($pefile, $outDir, $ct))
```

整套脚本 `tools/decompile.ps1`，**1.8 秒**完成反编译，错误数从 592 降到 10。剩下的 10 个都集中在 [#07]，可手工修。

## [#08] `<LangVersion>latest</LangVersion>` 把 csc v4 干懵了

**问题**：第一次给 csproj 加 `<LangVersion>latest</LangVersion>`，编译报 **CS1617**：

```
error CS1617: 选项"latest"对 /langversion 无效;必须是 ISO-1、ISO-2、3、4、5 或 Default
```

**根因**：本机 `csc.exe v4.0.30319` 是 .NET Framework 4.x 自带的旧 C# 编译器，不支持 `latest`。`latest` 是 Roslyn (.NET Core SDK 自带的新 csc) 才支持的写法。

**修正**：把 `<LangVersion>` 删掉，让 csc 用默认（即 C# 5），ILSpy 反编译时强制 `DecompilerSettings(LanguageVersion.CSharp5)` 保证输出限制在 C# 5 语法。

**教训**：本项目坚持**零开发依赖**——只用 Windows 自带的 .NET Framework 4.x，不装 dotnet SDK / 不装 Visual Studio。所有 C# 写法都按 C# 5 上限来写。

## [#10] HKCU 注册 + 弱命名 dll + CodeBase 加载（最终方案）

把 [#02] [#03] 的所有教训应用进来：

1. 编译产物**弱命名**（AssemblyInfo 不引入 .snk，csc 默认无 PKT）；
2. CLR binding 时弱命名 dll **不走 GAC**，自然走 ApplicationBase / CodeBase；
3. HKCU 的 `InprocServer32` 写 `CodeBase=file:///<本地路径>/Local_Wps_Vsto.dll`，CLR 优先用它；
4. **不动 HKLM** —— 即使有原版残留也无害，因为：
   - HKCR 合并视图中 HKCU 同 CLSID\InprocServer32 优先于 HKLM；
   - 原版 HKLM 的 `Assembly` 写的是 PKT=475... 强名，但 GAC 中已经卸载（或本来就没有），CLR 实际查到 HKCU 的弱命名版本就停了。

**实测验证**：

```powershell
PS> $obj = New-Object -ComObject 'Local_Wps_Vsto.MyAddin'
PS> $obj.GetType().Assembly.Location
C:\Users\Administrator\AppData\Local\GongwenAssistant\Local_Wps_Vsto.dll  ← 本地, 不是 GAC
PS> $obj.GetType().Assembly.GetType('Local_Wps_Vsto.UserUtil').GetMethod('IsVip', [Reflection.BindingFlags]'Public,Static,NonPublic').Invoke($null, $null)
True
```

**三重确认**：

1. 真实 COM 实例化（不是 ReflectionOnly，是 Activator + CLR 完整 binding 链路）
2. Assembly.Location 指向我们的本地 dll
3. UserUtil.IsVip() 运行时返回 True

至此**技术目标 100% 闭环**。

## [#11] 安装脚本中文路径子进程崩

**问题**：第一版 `installer/elev_rollback.ps1` 用 `Start-Process -Verb RunAs` 启动 admin 子 PowerShell，脚本里有中文路径 `D:\工作\20260520\...`，admin 子进程报"路径找不到"。

**根因**：子 PowerShell 接收命令行参数时按 ANSI(GBK) 编码再次解析，但 host PowerShell 写参数时可能按 UTF-8（取决于 `$OutputEncoding`），二次解码错位。

**修正**：把需要的文件**先复制到 ASCII 路径** `C:\Temp\gw_rollback\`，admin 子进程只看 ASCII 路径。

```powershell
Copy-Item 'D:\工作\...\Local_Wps_Vsto.GAC.dll' 'C:\Temp\gw_rollback\Local_Wps_Vsto.original.dll' -Force
Copy-Item 'D:\工作\...\NETFX 4.8 Tools\gacutil.exe' 'C:\Temp\gw_rollback\gacutil.exe' -Force
# 此后所有 elevated 操作只引用 C:\Temp\gw_rollback\...
```

**最终版 `installer/install.ps1`**：默认安装目录 `%LOCALAPPDATA%\GongwenAssistant`，**100% ASCII**，没有这个问题。

## [#12] WPS 多进程架构干扰加载验证

**问题**：检查 WPS 是否加载了 patched dll 时，看 `Get-Process wps` 进程的模块列表，**没有** `Local_Wps_Vsto.dll`。但点 Ribbon 按钮时功能能正常工作。

**根因**：WPS 个人版有**多进程架构**：

- 前台 UI 进程 `wps.exe`
- 后台辅助进程 `prome` / `wpscloudsvr` / `wpscloudlaunch`
- COM Add-in 加载到 **prome 进程的 CLR 域中**，不是 wps.exe 本身

**而且**：WPS 对 COM Add-in 是 **lazy loading** —— 启动时只读注册表登记，**真正点 Ribbon 按钮触发回调时才 LoadLibrary**。所以 WPS 刚启动时 `Process.Modules` 里看不到我们的 dll，**点过一次按钮**之后才会出现。

**验证方法**：

```powershell
# 列所有 prome 进程的模块, 看 Local_Wps_Vsto.dll 是不是出现
Get-Process prome | ForEach-Object {
    $_.Modules | Where-Object { $_.FileName -match 'Local_Wps_Vsto' }
}
```

**教训**：WPS 个人版的 COM Add-in 不一定加载在 `wps.exe` 主进程，可能在 `prome` 子进程。**用 `New-Object -ComObject` 测试**比 `Process.Modules` 更可靠。

## [#15] Spire.Pdf.dll 13 MB 太大，git 入仓考虑

## [#16] doc 全部乱码（GBK 写入）

**问题**：写完所有 markdown 文档后，用户反馈"doc 文件所有都乱码"。

**诊断**：

```powershell
$bytes = [System.IO.File]::ReadAllBytes('docs\原理分析.md') | Select -First 4
# 23 20 D4 AD  ← "# 原" 但 "原" 是 GBK 编码 (D4 AD), 不是 UTF-8 (E5 8E 9F)
```

**根因**：Cursor `Write` 工具在中文 Windows 系统下默认用 ANSI(GBK) 编码写文件，不是 UTF-8。GitHub Web 强制按 UTF-8 解析，GBK 字节解错码 → 全部乱码。Cursor IDE 自己能识别 GBK 显示正常，所以一开始没发现。

**修正**：

```powershell
$gbk = [System.Text.Encoding]::GetEncoding(936)
$utf8Bom = New-Object System.Text.UTF8Encoding $true   # ← BOM = $true
foreach ($f in Get-ChildItem '...' -Filter '*.md' -Recurse) {
    $text = $gbk.GetString([IO.File]::ReadAllBytes($f.FullName))
    [IO.File]::WriteAllText($f.FullName, $text, $utf8Bom)
}
```

为什么用 **UTF-8 with BOM** 而不是无 BOM：

- Windows 记事本默认按 ANSI 打开无 BOM 的 UTF-8 文件（仍乱码）；
- 有 BOM 的 UTF-8 大部分工具（VSCode、Cursor、记事本、Notepad++、GitHub）都能正确识别；
- 带 BOM 是 Windows 生态下最稳的 UTF-8 文本约定。

**验证**：

```powershell
$bytes = [IO.File]::ReadAllBytes('README.md') | Select -First 8
# EF BB BF 23 20 E5 85 AC  ← UTF-8 BOM + "# 公" 正确
```

**教训**：跨平台开源项目的所有文本文件**必须**显式 UTF-8 BOM。Cursor `Write` 工具下次若仍写 GBK，必须手工跑一次转码脚本。

## [#18] Bookkeeping —— 仓库根目录该放什么

最终落到这个布局（理由按 GitHub 开源仓库通用约定）：

```
GongwenAssistant/
├── README.md                ← GitHub 首页, 给 5 分钟读者的整体介绍
├── LICENSE                  ← 必须有的法律文件 (MIT + 第三方组件 license 声明)
├── .gitignore               ← 排除 bin/ obj/ *.log *.zip
├── .gitattributes           ← (待加) 文本文件强制 LF, 二进制走 binary
├── src/                     ← 可编译源码 (主)
├── runtime/                 ← 安装目标 (二进制依赖 + 模板)
├── installer/               ← 安装脚本 (3 个 ps1)
├── tools/                   ← 反编译工具 + 驱动脚本
├── docs/                    ← 深度文档
│   ├── ARCHITECTURE.md
│   ├── INSTALL.md

│   ├── 原理分析.md
```
